# REQ-0002：为 workflow Skills 增加可审计的自动编排能力

## 原始需求

> 想结合 `.workflow/` 做得更「企业级、可审计」一些。
>
> 希望能有某种方式自动编排目前的 Skills，不用手动选择；同时支持手动调用 Skill 作为 override。

## 背景与目标

REQ-0001 已经为每个 workflow Skill 增加了「执行契约」（Load / Assert / Validate），确保单个 Skill 被调用时会强制读取上下文、校验状态、验证文件。但这些契约仍然依赖用户手动选择调用哪个 Skill。

本需求目标是：

1. 建立一个 **Skill Registry**，注册可用 Skills 的输入输出契约、副作用、安全级别。
2. 实现一个轻量 **编排引擎**，能够根据 `.workflow/current.md` 和当前 REQ/plan 推荐或执行下一步 Skill。
3. 建立 **审计日志**，记录每次编排的触发原因、执行步骤、人工确认点。
4. 保持用户控制：写文件、git 操作、外部 API 调用等高风险动作必须人工确认。
5. 触发方式以显式命令 `/orchestrate <req-id>` 或 `workflow-template --orchestrate <req-id>` 为主，避免像 Superpowers 那样无差别自动触发。

成功后，用户不再需要记住该调用哪个 Skill，同时每一次自动调用都有据可查、有规则可依。

## 关联范围

### 相关模块

- `bin/workflow.js`：新增 `--orchestrate` 命令。
- `lib/orchestrator.js`（新建）：编排引擎核心。
- `.workflow/orchestration/skill-registry.yaml`：Skill 注册表。
- `.workflow/orchestration/execution-policies.yaml`：自动执行 / 人工确认策略。
- `.workflow/audit/executions/`：审计日志目录。

### 相关文件 / 接口 / 数据表

- `.workflow/current.md`
- `.workflow/requirements/REQ-*.md`
- `.workflow/plans/REQ-*-plan.md`
- `.workflow/implementations/REQ-*-implementation.md`
- REQ-0001 中已实现的 `--assert-status` 与 `--validate`

### 关联需求 / 能力规格

- 参考需求：REQ-0001（Skill 执行契约）
- 复用能力：REQ-0001 暴露的 `--assert-status` 与 `--validate`

## 业务规则

| 场景             | 规则                                                                                       | 备注                       |
| ---------------- | ------------------------------------------------------------------------------------------ | -------------------------- |
| 启动编排         | 用户必须显式调用 `/orchestrate <req-id>` 或 `workflow-template --orchestrate <req-id>`     | 保证触发意图可审计         |
| 编排器读取上下文 | 必须读取 `.workflow/project.md`、`.workflow/current.md`、目标 `REQ-*`、plan/implementation | 复用 REQ-0001 的 Load 契约 |
| 推荐下一步 Skill | 基于当前需求状态、plan 步骤、skill-registry 的触发条件                                     | 推荐不等于自动执行         |
| 高风险 Skill     | 写文件、git commit/push、外部 API 调用必须 `human_confirm: true`                           | 人工确认记录写入审计日志   |
| 状态推进         | 推进前必须调用 `workflow-template --assert-status`                                         | 复用 REQ-0001              |
| 文件修改后       | 必须调用 `workflow-template --validate`                                                    | 复用 REQ-0001              |
| 审计日志         | 每次编排执行必须生成一条 execution 记录                                                    | 见执行策略                 |

## 边界条件

### 做什么

- MVP 阶段实现 Skill Registry + 审计日志。
- 新增 CLI 命令触发编排会话。
- 编排器能读取 current.md 和 REQ/plan，推荐下一步 Skill。
- 定义哪些 Skill 自动执行、哪些必须人工确认。
- 审计日志包含触发原因、步骤、Skill、确认记录。

### 不做什么

- MVP 不做自然语言自动路由（不模仿 Superpowers 的全自动触发）。
- 不实现完整 DAG 工作流引擎。
- 不修改现有 Skill 的安装路径或文件格式。
- 不改动 REQ-0001 已实现的 validator 核心逻辑。

### 异常情况

- 目标 REQ 不存在：编排器报错并退出。
- 当前无 plan：编排器建议先调用 `workflow-plan`。
- Skill 执行失败：记录失败状态，停止编排，不自动重试（MVP）。
- 用户拒绝确认：记录拒绝原因，停止当前步骤。

## 需求确认

### 术语定义

| 术语                 | 定义                                                         |
| -------------------- | ------------------------------------------------------------ |
| Skill Registry       | 注册可用 Skills 的元数据文件，包含契约、触发条件、安全级别。 |
| Orchestration Engine | 根据当前上下文和注册表推荐/执行 Skill 的轻量引擎。           |
| Execution Policy     | 定义某个 Skill 是自动执行还是必须人工确认。                  |
| Audit Log            | 记录编排执行全过程的结构化日志。                             |

### 数据流 / 调用链路

```text
用户：/orchestrate REQ-0002
  -> CLI 解析 --orchestrate REQ-0002
  -> 编排器读取 .workflow/current.md
  -> 编排器读取 requirements/REQ-0002.md、plans/REQ-0002-plan.md
  -> 编排器根据状态推断下一步 Skill
  -> 如果下一步 Skill 是 read-only：自动调用，记录审计日志
  -> 如果下一步 Skill 有 write-side-effect：暂停，请求人工确认
  -> 用户确认后调用 Skill
  -> Skill 执行过程中调用 --assert-status 与 --validate
  -> 结果写入 implementations/REQ-0002-implementation.md
  -> 编排器生成最终 audit log
```

### 权限 / 安全 / 数据一致性

- 权限：编排器只读取 `.workflow/` 和运行本地 CLI。
- 安全：高风险操作必须人工确认，审计日志记录确认人（在 Claude Code 场景下为用户本人）。
- 数据一致性：任何 `.workflow/` 文件修改后必须 `--validate`；状态推进前必须 `--assert-status`。

## 测试用例

### 正常场景

- `/orchestrate REQ-0002` 启动编排，正确读取上下文。
- 推荐 read-only Skill 时自动执行并记录日志。
- 推荐 write-side-effect Skill 时暂停等待确认。

### 边界场景

- 当前 REQ 无 plan：编排器建议先 plan。
- Skill 不在 registry 中：编排器提示未注册，不自动调用。

### 异常场景

- 目标 REQ 不存在：编排器报错退出码 1。
- 用户拒绝确认：记录拒绝并停止，不继续执行。
