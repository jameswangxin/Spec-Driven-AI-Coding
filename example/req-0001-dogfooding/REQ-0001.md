# REQ-0001：工作流规则可执行化与真源记录

## 原始需求

Codex adversarial review 指出两个必须修复的高优先级问题：

1. 新增 `.workflow/index.md` 与 `.workflow/current.md` 缺少受管理标记，导致 `node bin/workflow.js --sync-index` / `--sync-current` 会直接失败（`Missing workflow marker pair`）。
2. 本次“规则可执行化”变更本身没有自己的 `REQ-*` / `plan` / `implementation` 真源，与 `.workflow/` 作为唯一真源的约定冲突。

## 背景与目标

本仓库已经实现了 `lib/validator.js`、`bin/workflow.js` 的校验与同步能力，但 `.workflow/` 目录仍是初始化后的占位状态。为了让工作流工具能够管理自己的索引和当前上下文，并让本次变更可被审计、追溯，需要：

- 为 `.workflow/index.md` 的 Active Work、Capability Specs 表以及 `.workflow/current.md` 的 Current Requirement 区块补全 `workflow:*:start/end` 受管理标记。
- 创建 REQ-0001 及对应的 plan、implementation，使 current/index 指向它。
- 将原位于 `docs/superpowers/plans/2026-06-20-workflow-enforcement.md` 的有效计划内容迁入 `.workflow/plans/REQ-0001-plan.md`。
- 实际运行 `validate`、`sync-index`、`sync-current` 与 `npm test`，并将结果写入 implementation。

## 关联范围

### 相关模块

- `.workflow/index.md`
- `.workflow/current.md`
- `.workflow/requirements/`
- `.workflow/plans/`
- `.workflow/implementations/`
- `docs/superpowers/plans/2026-06-20-workflow-enforcement.md`

### 相关文件 / 接口 / 数据表

- `lib/validator.js` 中的 `syncIndex`、`syncCurrent`、`validateWorkflow`。
- `bin/workflow.js` 中的 `--validate`、`--sync-index`、`--sync-current`。

### 关联需求 / 能力规格

- 参考需求：无。
- 复用能力：无（暂不创建 CAP，避免空引用导致校验失败）。

## 业务规则

| 场景         | 规则                                                                                | 备注                                                 |
| ------------ | ----------------------------------------------------------------------------------- | ---------------------------------------------------- |
| index 同步   | `syncIndex` 仅替换 `workflow:active-work` 与 `workflow:capabilities` 标记之间的内容 | 标记缺失会抛出 `Missing workflow marker pair`        |
| current 同步 | `syncCurrent` 仅替换 `workflow:current` 标记之间的内容                              | 显式传入 ID 时不校验状态；省略 ID 时要求唯一活动需求 |
| 计划真源     | Superpowers 计划内容必须转写到 `.workflow/plans/REQ-xxxx-plan.md`                   | 见 `.workflow/integrations/superpowers.md`           |
| 验证记录     | 所有验证命令、结果与偏差必须写入 implementation                                     | 未记录验证不得更新为 `verified`                      |

## 边界条件

### 做什么

- 补全受管理标记。
- 创建 REQ-0001、plan、implementation。
- 迁移有效计划内容。
- 运行并记录 CLI 验证与测试。

### 不做什么

- 不修改全局 Superpowers Skill。
- 不修改 `lib/validator.js` 或 `bin/workflow.js` 的已有实现逻辑。
- 不创建新的能力规格（CAP）。

### 异常情况

- 如果 `npm test` 失败，必须将需求状态保持或回退到 `implemented`，记录失败原因与残余风险。
- 如果 `docs/superpowers/plans/2026-06-20-workflow-enforcement.md` 删除后影响其他引用，需在 implementation 中记录。

## 需求确认

### 术语定义

| 术语       | 定义                                                                                |
| ---------- | ----------------------------------------------------------------------------------- |
| 受管理标记 | `<!-- workflow:<name>:start -->` 与 `<!-- workflow:<name>:end -->` 之间的可替换区块 |
| 真源       | `.workflow/` 作为需求、方案、实施和验证的唯一权威记录                               |

### 数据流 / 调用链路

```text
用户请求修复
  -> 修改 .workflow/index.md / current.md 的受管理标记
  -> 创建 REQ-0001 / plan / implementation
  -> 运行 node bin/workflow.js --validate
  -> 运行 node bin/workflow.js --sync-index / --sync-current REQ-0001
  -> 运行 npm test
  -> 更新 implementation 测试记录
  -> 如全部通过，更新 REQ-0001 状态为 verified
```

### 权限 / 安全 / 数据一致性

- 权限：仅修改 `.workflow/` 与已存在的 `docs/superpowers/plans/2026-06-20-workflow-enforcement.md`。
- 安全：无敏感数据。
- 数据一致性：确保 requirement frontmatter 的 `status` 与 `history` 末项一致；确保 `updated_at` 不早于最新 `history` 日期。

## 测试用例

### 正常场景

- 运行 `node bin/workflow.js --validate`，预期返回 `Workflow validation passed.`。
- 运行 `node bin/workflow.js --sync-index`，预期重写 index 中受管理区块。
- 运行 `node bin/workflow.js --sync-current REQ-0001`，预期重写 current 中受管理区块。
- 运行 `npm test`，预期全部测试通过。

### 边界场景

- 在 `capabilities: []` 的情况下，`validate` 不应报 `WF_CAPABILITY_MISSING`。

### 异常场景

- 如果 index/current 仍缺少标记，同步命令应报错；本需求通过验收即排除该异常。
