# REQ-0001：Skills 仅依赖 Markdown 提示，缺少可执行约束

## 原始需求

> 1.7 Skills 只是 Markdown 提示，没有可执行约束
>
> skills/workflow-\*/SKILL.md 写得清楚，但它们本质是“请 Agent 遵守的说明书”。Agent 是否真读、是否真按步骤执行，完全依赖模型自律。这和安装器/CLI 提供的文件级、状态级校验是脱节的。
>
> 真正好用的工作流 Skill 应该：
>
> - 在调用时自动读取 .workflow/current.md、相关 REQ-\*、plan、implementation。
> - 在修改文件后自动调用 --validate。
> - 在状态推进前检查前置条件（例如 workflow-exec 应调用 assertStatus 确保状态是 planned）。
>
> 目前这些能力都在 lib/validator.js 里，但 Skills 没有调用它们。

## 背景与目标

本仓库是规格驱动 AI 编码工作流脚手架，`.workflow/` 是需求、方案、实施、验证的真源。CLI (`bin/workflow.js`) 和 `lib/validator.js` 已经提供了 `--validate`、`--sync-index`、`--sync-current` 以及 `assertStatus` 等能力，但安装的 Skills（`workflow-new`、`workflow-confirm`、`workflow-plan`、`workflow-exec`、`workflow-check`、`workflow-archive`）只是 Markdown 说明，Agent 是否执行校验完全依赖自律。

本需求目标：

1. 让 Skill 调用时**强制**加载 `.workflow/current.md`、相关 `REQ-*`、plan、implementation 等上下文。
2. 让 Skill 在修改 `.workflow/` 文件后**强制**运行校验。
3. 让 Skill 在状态推进前**强制**检查前置状态（如 `workflow-exec` 要求 `planned`）。
4. 将 `lib/validator.js` 中的 `assertStatus` 等能力通过 CLI 暴露给 Skills。
5. 保持 Skill 仍以 Markdown 形式存在（兼容现有安装器），但内部包含可执行的命令约束。

## 关联范围

### 相关模块

- `bin/workflow.js`：CLI 入口，需要新增 `--assert-status` 等命令。
- `lib/validator.js`：已包含 `assertStatus`、`validateWorkflow`、`syncIndex`、`syncCurrent`。
- `lib/installer.js`：安装 Skills 到目标项目，可能需要安装辅助脚本或更新 settings。
- `skills/workflow-*/SKILL.md`：需要改写，加入强制工具调用指令。
- `test/validator.test.js`：新增 `--assert-status` 与 Skill 集成相关测试。

### 相关文件 / 接口 / 数据表

- `.workflow/current.md`、`.workflow/index.md`
- `.workflow/requirements/REQ-*.md`
- `.workflow/plans/REQ-*-plan.md`
- `.workflow/implementations/REQ-*-implementation.md`

### 关联需求 / 能力规格

- 复用能力：无。

## 业务规则

| 场景                         | 规则                                                                                               | 备注                                      |
| ---------------------------- | -------------------------------------------------------------------------------------------------- | ----------------------------------------- |
| Skill 被调用                 | 必须先读取 `.workflow/project.md`、`.workflow/current.md`、目标 `REQ-*` 及相关 plan/implementation | 不再依赖 Agent 自律                       |
| Skill 修改 `.workflow/` 文件 | 修改后必须运行 `workflow-template --validate`                                                      | 文件级校验                                |
| Skill 推进状态               | 推进前必须调用 `workflow-template --assert-status REQ-xxxx --status <allowed>`                     | 状态级校验                                |
| CLI 暴露 assertStatus        | `--assert-status REQ-xxxx --status planned` 成功退出码 0，失败退出码 1                             | 供 Skill 调用                             |
| 多状态允许                   | `--status` 支持多次传入，以允许列表方式工作                                                        | 例如 `--status accepted --status planned` |

## 边界条件

### 做什么

- 扩展 CLI 暴露 `assertStatus`。
- 改写所有 workflow Skills 的 SKILL.md，加入强制读取、校验、状态断言指令。
- 让安装器继续只复制 SKILL.md（不引入新文件类型），保持 Skill 兼容性。
- 更新测试覆盖新增 CLI 行为。

### 不做什么

- 不修改 Skill 的安装路径或文件格式（仍为 `.codex/skills/<name>/SKILL.md` 与 `.claude/skills/<name>/SKILL.md`）。
- 不引入新的运行时 hook 或 Agent harness 配置。
- 不改动 validator 核心校验逻辑（只暴露接口）。

### 异常情况

- 目标项目未初始化 `.workflow/`：Skill 指令中已要求先读取 project.md，未初始化时 Agent 会自然失败。
- `workflow-template` 命令不存在：Skill 指令中可提示使用 `npx workflow-template` 或本地 `node ./node_modules/.bin/workflow-template`。
- 校验失败：Skill 必须停止并报告错误，不得继续推进状态。

## 需求确认

### 术语定义

| 术语         | 定义                                                                                                                     |
| ------------ | ------------------------------------------------------------------------------------------------------------------------ |
| Skill        | Claude Code / Codex 读取的 Markdown 系统提示，位于 `.codex/skills/<name>/SKILL.md` 或 `.claude/skills/<name>/SKILL.md`。 |
| 可执行约束   | Skill 文本中包含明确的工具调用指令（如“必须使用 Bash 运行 ……”），使 Agent 在调用时实际执行校验。                         |
| assertStatus | `lib/validator.js` 导出的函数，检查需求当前状态是否在允许列表中。                                                        |

### 数据流 / 调用链路

```text
User invokes Skill
  -> Skill prompt instructs: read current.md + REQ-* + plan/implementation
  -> Agent uses Read tool
  -> Skill prompt instructs: run workflow-template --assert-status ...
  -> Agent uses Bash tool
  -> Skill prompt instructs: perform work
  -> Skill prompt instructs: run workflow-template --validate
  -> Agent uses Bash tool
  -> Skill prompt instructs: sync index/current if needed
```

### 权限 / 安全 / 数据一致性

- Skill 只读取 `.workflow/` 与运行本地 CLI，不外发数据。
- 校验命令以只读方式扫描文件（`--validate` 不修改文件）。
- 状态断言失败时不允许修改文件。

## 测试用例

### 正常场景

- `workflow-template --assert-status REQ-0001 --status accepted --status planned` 在 `accepted` 状态成功。
- Skill 指令文本中包含 `--validate` 与 `--assert-status` 调用要求。

### 边界场景

- `--assert-status` 指定多个 `--status` 参数时，任一匹配即成功。
- `--assert-status` 未提供 `--status` 时报错。

### 异常场景

- `--assert-status` 目标 REQ 不存在时报 `WF_REQUIREMENT_MISSING`。
- `--assert-status` 状态不匹配时报 `WF_STATUS_NOT_ALLOWED`。
